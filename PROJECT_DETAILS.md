# AttendX: Smart Attendance System - Full Project Documentation

AttendX is a comprehensive, full-stack biometric attendance management system. It leverages Artificial Intelligence (AI), Geolocation (GPS), and Network Fingerprinting (WiFi) to ensure that attendance marking is secure, fraud-proof, and real-time.

---

## 1. System Architecture
The system uses a modern **decoupled architecture**, separating the client-side interaction from the server-side processing.

*   **Frontend (Android Application):** A high-performance mobile app built with Kotlin and Jetpack Compose. It handles user interactions, biometric data capture, and location-based scanning.
*   **Backend (RESTful API):** A Python-based server using **FastAPI**. It acts as the orchestration layer, handling authentication, business logic, and AI-driven face verification.
*   **Database (Hybrid Cloud Storage):**
    *   **Supabase (PostgreSQL):** Primary cloud database for relational data, schedules, and permanent logs.
    *   **SQLAlchemy ORM:** Used to bridge the Python code with the database for easy data manipulation.
    *   **SQLite Fallback:** A local file-based database that ensures system availability even when the cloud database is unreachable.
*   **Storage Layer:** Face images are stored both as **binary data (LargeBinary/BYTEA)** in the database and as **physical files** on the server for high-speed AI processing.

---

## 2. Technical Stack

### **Mobile Layer (Android/Kotlin)**
*   **Jetpack Compose:** Used for building a reactive, modern UI with a focus on dark-mode aesthetics.
*   **Retrofit & OkHttp:** Handles all network communication with the FastAPI backend, including multi-part file uploads for face images.
*   **Google ML Kit:** 
    *   *Face Detection:* To ensure a human face is present before capturing a selfie.
    *   *Barcode Scanning:* To parse unique, encrypted QR codes generated by the faculty.
*   **CameraX API:** Manages the device camera for both front (selfie) and back (QR scan) modes.
*   **Fused Location Provider:** Captures precise GPS coordinates for geofencing.

### **Backend Layer (Python/FastAPI)**
*   **FastAPI:** Chosen for its asynchronous capabilities (high concurrency) and automatic documentation generation.
*   **DeepFace (AI Core):** A state-of-the-art deep learning library that wraps models like VGG-Face, FaceNet, and OpenFace. It performs face verification by comparing a live selfie against a registered master image.
*   **OpenCV:** Used for pre-processing images, such as resizing, normalizing, and converting color spaces for the AI model.
*   **Uvicorn:** A lightning-fast ASGI server implementation for Python.

### **Security & Infrastructure**
*   **Network Fingerprinting:** Each classroom is mapped to a specific **WiFi BSSID (MAC Address)** and **SSID**. The system prevents attendance marking if the student isn't connected to the correct classroom WiFi.
*   **Dynamic QR Tokens:** QR codes are not static; they contain a session-specific token that expires within minutes to prevent students from sharing photos of the QR code.
*   **Device ID Binding:** Students are bound to their hardware ID to prevent one student from logging in on multiple phones for their friends.

## 3. Database Schema (Entities)

*   **App Users (`app_users`):** Central authentication table (Username, Email, Role, Password Hash).
*   **Students (`app_students`):** Stores academic details (Roll No, Branch, Year) and biometric data (Face Embedding, Face Image).
*   **Teachers (`app_teachers`):** Stores faculty profiles and departmental information.
*   **Classrooms (`classrooms`):** Maps physical rooms to WiFi BSSID/SSID and GPS center points.
*   **Subjects & Schedules:** Defines which faculty teaches which course and at what time.
*   **Attendance Sessions:** Real-time table that tracks active classes and their specific QR tokens.
*   **Attendance Records:** The final log of verified attendance, including timestamps, location data, and verification status.

## 4. Operational Workflow

### **Phase 1: Session Initialization (Faculty)**
1.  Faculty logs in and selects a scheduled class.
2.  The system validates the time and location.
3.  A unique `session_id` is created, and a QR code is generated.
4.  The faculty starts the session, and the app begins live-polling for attendance updates.

### **Phase 2: Verification Loop (Student)**
1.  **Identity:** Student logs into their bound device.
2.  **Location:** The app verifies the student is connected to the classroom WiFi.
3.  **Proximity:** Student scans the faculty's dynamic QR code.
4.  **Biometrics:** Student takes a live selfie. The backend compares this against the "Master Face" in the database.
5.  **Confirmation:** If all three layers (QR + WiFi + Face) match, the `attendance_records` table is updated.

### **Phase 3: Real-Time Reporting**
1.  The faculty sees student names appearing on their screen in real-time.
2.  Once the session ends, the system automatically marks non-participating students as "Absent" and generates a summary report.

## 6. Key Benefits
*   **Zero Proxy:** The combination of WiFi proximity and AI Face Verification makes it nearly impossible for a student to mark attendance for an absent friend.
*   **Scalability:** Uses Supabase for cloud-scale data management and FastAPI for high-throughput request handling.
*   **User Experience:** Automated pre-filling of schedules ensures that faculty can start a session in just two clicks.
